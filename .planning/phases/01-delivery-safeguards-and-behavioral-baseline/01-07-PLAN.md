<threat_model>
## Trust Boundaries
| Boundary | Description |
|---|---|
| Source tree/inventory → validator | Untrusted paths, anchors, JSON keys, and ownership references enter tooling. |
| Canonical JSON → generated Markdown | Generated output could conceal or reorder capability gaps. |

## STRIDE Threat Register
| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|---|---|---|---|---|---|
| T-01-01 | Tampering | source paths | high | mitigate | Reject absolute/`..`, normalize `/`, require existing in-repo paths/anchors. |
| T-01-02 | Tampering | schema/prototype keys | high | mitigate | Ajv strict mode, bounded enums, and `additionalProperties:false`. |
| T-01-03 | Tampering | generated report | high | mitigate | JSON canonical; deterministic byte-freshness check blocks hand edits. |
| T-01-04 | Repudiation | ownership/evidence | medium | mitigate | Every record requires explicit requirement, phase, source, and evidence. |
| T-01-05 | Elevation/Disclosure | PR boundary | high | mitigate | Plan 01-09 uses read-only PR permissions and safe artifacts. |
| T-01-06 | Disclosure | inventory content | medium | mitigate | Record capability contracts, never secrets or credential values. |
| T-01-07 | Tampering | Git refs/commands | high | mitigate | Version tooling uses normalized paths and argument arrays. |
| T-01-SC | Tampering | Ajv dependency | high | mitigate | Exact approved lockfile and `npm ci`. |
</threat_model>
